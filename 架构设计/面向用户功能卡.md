# 多智能体脚本（Agents）——面向用户功能卡

> **运行环境**：本脚本是一个运行在 SillyTavern（酒馆）助手（一个为 SillyTavern 提供扩展能力的浏览器插件）之上的 JS 脚本。  
> **核心功能**：允许用户配置多个「智能体」，每个智能体对应一次独立的模型调用步骤。用户发送一条消息后，脚本按智能体排列顺序依次执行生成，每个智能体可使用不同的 API、模型、参数和提示词组合。  
> **姊妹脚本**：本脚本与同项目下的「变量更新脚本（VarUpdate）」配合使用。VarUpdate 负责解析 AI 输出中的变量更新指令并写入酒馆助手的变量系统（酒馆助手插件提供的原生键值存储功能，所有基于酒馆助手构建的脚本均可读写）。两个脚本通过事件通信松耦合，可独立运行，也可协同工作。

---

## 一、智能体配置

每个智能体代表流水线中的一个独立生成步骤。智能体以大写英文字母标识排序位（A、B、C、D、E……），用户可自由增删、排序。

### A-1 智能体定义与管理

- 用户可以创建、编辑、复制、删除智能体。
- 智能体以大写英文字母标识排序位（A、B、C……），而非数字。用户可拖动或手动调整排序，字母标识随排序自动重新分配。
- 每个智能体拥有独立的：
  - **名称**（用户自定义，用于 UI 展示和辨识用途）。
  - **字母标识**（自动分配，用于排序和执行状态指示）。
  - **备注**（可选，用户可填写该智能体的用途说明，仅用于自我提示，不参与任何逻辑）。
- 智能体的启用/禁用状态不是智能体自身的属性，而是**方案级属性**——同一个智能体在不同方案中可以有不同的启用状态。启用/禁用通过 UI 面板的工作区/待命区机制管理（见 I-3），状态随方案保存（见 F-1）。

### A-2 API 格式选择

- 采用与酒馆原生设置页面一致的交互流程：用户首先选择 API 格式（如 OpenAI、Claude、Google AI Studio 等），选定后界面动态展示该格式对应的全部可配置项。
- API 格式列表在运行时从酒馆核心动态读取（`chat_completion_sources`），不由脚本硬编码维护。酒馆版本更新新增 API 格式时，脚本自动跟进。

### A-3 连接参数配置

- 每个智能体独立配置以下连接信息：
  - **API 地址（URL）**：该智能体请求的目标服务器地址。
  - **API Key**：该智能体使用的身份凭证。
  - **模型名称**：手动填写或从模型列表中选择（见 A-4）。

### A-4 一键获取模型列表

- 当用户已填写 API 地址和 API Key 后，可点击按钮向该地址发送模型列表请求。
- 返回的可用模型以下拉列表形式展示，用户可从中选择，省去手动输入模型名称的麻烦。
- 若请求失败（地址错误、Key 无效、服务端不支持列表接口等），给出明确错误提示。

### A-5 模型参数配置

脚本自建参数配置面板。参数的可见性由当前智能体选择的 API 格式决定——脚本内部维护一份 **API 格式→支持参数的映射表**，仅显示当前格式支持的参数项。每个参数具有预设的默认值，显示即生效，其值始终包含在 API 请求中。数值型参数提供**滑动条**控件，用户可通过拖动滑块或直接输入数字进行调整。脚本不读取、不依赖、不引用酒馆原生预设中的任何参数值，每个智能体的参数配置完全自包含。

参数分为**通用配置**和**小众配置**两组：

#### A-5a 通用配置

映射表中几乎所有 API 格式都包含的参数：

| 参数 | 控件 | 默认值 | 说明 |
|---|---|---|---|
| 流式传输 | 开关 | 开启 | 是否启用流式传输（streaming） |
| 工具调用 | 开关（置灰） | 关闭 | 第一版界面置灰标注「即将支持」，底层预留配置项 |
| 传回推理过程 | 开关 | 关闭 | 是否要求 API 在响应中包含推理/思维链内容。关闭时脚本剥离推理部分仅保留正式回复 |
| 推理强度 | 下拉选择 | medium | 控制模型的推理深度（reasoning effort） |
| 最大回复长度（max_tokens） | 数值输入 | 4096 | 单次回复的最大 Token 数 |
| temperature | 滑动条（0~2） | 1.0 | 采样温度 |
| top_p | 滑动条（0~1） | 1.0 | 核采样概率阈值 |

#### A-5b 小众配置

仅部分 API 格式支持的参数。脚本内部维护一份 **API 格式→支持参数的映射表**，根据当前智能体选择的 API 格式（A-2）动态显示对应的小众参数项。当 API 格式选为「自定义」时，所有小众参数全部显示。

| 参数 | 控件 | 默认值 | 支持的典型格式 |
|---|---|---|---|
| min_p | 滑动条（0~1） | 0.0 | OpenRouter、Chutes |
| top_k | 数值输入 | 0 | Claude、OpenRouter、Google/VertexAI、Cohere、Perplexity、ElectronHub、Chutes |
| frequency_penalty（频率惩罚） | 滑动条（-2~2） | 0.0 | 大部分格式（xAI 部分模型除外） |
| presence_penalty（存在惩罚） | 滑动条（-2~2） | 0.0 | 大部分格式（xAI 部分模型、ZAI 除外） |

> 映射表中的具体条目将在统一定义阶段确定完整清单。

## 二、标签与提示词注入

脚本不集中管理提示词内容。提示词的创建和编辑完全在酒馆原生的预设（Preset）和世界书（World Info / Lorebook）体系中完成。脚本通过一套基于 `[]` 定界符的**标签语法**来控制哪些条目在哪些步骤中、在什么条件下被注入。

### B-1 标签语法概述

- `[]` 是标签的统一定界符，属于语法的固定组成部分，不由用户自定义。
- 标签写在预设条目的标题或世界书条目的备注（comment）中。
- **标签位置不限**：标签可以放在标题/备注文本的任何位置——开头、中间、末尾均可生效。脚本通过全文匹配检测标签是否存在，不要求固定位置。
- 一个条目可以携带多个标签（多个 `[]` 并排），脚本依次解析每一个。

#### 标签类型的区分规则

解析器通过 `[]` 内部的首字符来判定标签类型：

| 首字符 | 类型 | 示例 |
|---|---|---|
| `"` 开头 | 变量条件标签 | `["HP" >= 60]`、`["地点" == "酒馆"]` |
| 除 `"` 以外的普通字符 | 智能体匹配标签或特殊标签 | `[创意写作]`、`[NonPrompts]`、`[All]` |
| `!` 后接 `"` | 变量条件的否定（对整个条件取反） | `[!"HP" >= 60]`（HP 不满足 ≥ 60 时为真） |
| `!` 后接普通字符 | 智能体排除标签 | `[!变量更新]` |

简而言之：**变量名始终用双引号包裹，智能体标签名和特殊标签名不使用引号**。这是解析器区分两者的唯一依据。

**同一个 `[]` 内不允许混合智能体标签和变量条件标签。** `|`（OR）只能连接同类型的项：要么全是智能体标签（如 `[创意写作|变量更新]`），要么全是变量条件（如 `["心情" == "开心"|"心情" == "高兴"]`）。跨类型的组合通过多个 `[]` 并排（AND）实现，例如 `[创意写作]["情境" == "战斗"]`。

#### 值的字面量语法

在变量条件标签中，比较值的书写规则如下：

| 类型 | 写法 | 示例 |
|---|---|---|
| 字符串 | 用双引号包裹 | `"酒馆"`、`"***火焰***"` |
| 数值 | 不加引号，直接书写 | `60`、`3.14`、`-1` |
| 布尔值 | 不加引号，使用 `true` / `false` | `true`、`false` |
| 变量引用 | `$"变量名"` | `$"最大HP"`、`$"敌人/防御力"` |

#### 逻辑组合规则

| 逻辑 | 语法 | 说明 |
|---|---|---|
| **OR（或）** | 同一个 `[]` 内用 `|` 分隔 | 须类型相同（同为智能体标签或同为变量检查标签），多项满足任一即可。例：`[创意写作|变量更新]`、`["心情"=="开心"|"心情"=="高兴"]` |
| **AND（与）** | 多个 `[]` 并排 | 全部满足才注入。例：`[创意写作]["情境" == "战斗"]` |
| **NOT（非）** | `[]` 内首位加 `!` | 对该标签/条件取反。例：`[!变量更新]`、`[!"HP" >= 60]` |

多个标签并排时，它们之间是 AND 关系——所有标签条件都满足时，该条目才会在当前步骤中被注入。AND、OR、NOT 三种逻辑可自由组合，实现完备的布尔逻辑。例如：

```text
[创意写作|变量更新][!"地点" == "虚空"]["好感度" >= 30]
```

含义：在「创意写作」或「变量更新」步骤中，且地点不是"虚空"，且好感度 ≥ 30 时，注入该条目。

### B-2 智能体匹配标签

用于控制条目在哪些智能体步骤中被注入。

- 每个智能体在配置时拥有一个唯一的标签名称（由用户自定义，如 `创意写作`、`变量更新`、`摘要生成`）。
- 在预设/世界书条目中写入 `[标签名]` 即声明该条目应在对应步骤中注入。

| 语法 | 含义 | 示例 |
|---|---|---|
| `[标签名]` | 在该智能体步骤中注入 | `[创意写作]` |
| `[标签A|标签B]` | 在 A 或 B 步骤中注入 | `[创意写作|变量更新]` |
| `[!标签名]` | 在该智能体步骤中**不**注入（其余步骤均注入） | `[!变量更新]` |
| `[!标签A|!标签B]` | 在 A 和 B 步骤中都不注入（其余均注入） | `[!变量更新|!摘要生成]` |

排除语法 `!` 的典型用例：某条世界书设定几乎所有步骤都需要，唯独变量更新步骤不需要。用 `[!变量更新]` 比逐一列举其余所有智能体标签方便得多。

### B-3 变量条件标签

用于根据酒馆助手变量系统中的变量值，动态决定条目是否注入。变量系统是酒馆助手插件的原生功能，所有基于酒馆助手构建的脚本（包括本脚本）都可以读取和使用其中的变量。

> **变量来源**：标签条件中引用的变量读取自最新一层消息的 `data` 字段（即 VarUpdate 脚本存储在 message 层的变量快照）。这与 VarUpdate 的插值宏 `{{message/data/...}}` 读取同一个数据源。

#### B-3a 基本比较

对变量值与字面量进行比较。语法为 `["变量名" 运算符 值]`。

| 运算符 | 含义 | 示例 |
|---|---|---|
| `==` | 宽松相等（不校验类型） | `["地点" == "酒馆"]` |
| `!=` | 宽松不等 | `["地点" != "酒馆"]` |
| `===` | 严格相等（校验类型） | `["HP" === 100]` |
| `!==` | 严格不等 | `["HP" !== "100"]`（排除字符串 "100"） |
| `>` | 大于 | `["好感度" > 0]` |
| `>=` | 大于等于 | `["好感度" >= 60]` |
| `<` | 小于 | `["好感度" < 80]` |
| `<=` | 小于等于 | `["好感度" <= 100]` |

数值比较（`>` `>=` `<` `<=`）适用于 number 类型的变量。字符串相等/不等（`==` `!=`）适用于任何类型。宽松比较（`==` `!=`）在类型不一致时尝试自动转换，严格比较（`===` `!==`）要求类型和值都匹配。

组合示例：`["好感度" >= 60]["好感度" < 80]至交好友` —— 好感度在 60（含）到 80（不含）之间时注入。

#### B-3b 通配符匹配

在任何涉及字符串值匹配的场景中，都可以使用 `*` 通配符进行模糊匹配。这包括：字符串比较（`==`、`!=`、`===`、`!==`）、数组含值（`∋`、`!∋`）、对象含键（`⊇`、`!⊇`）。

**通配符个数规则：**

- **1~2 个 `*`**：每个 `*` 严格匹配恰好 1 个字符。个数和位置都必须精确对应。
- **3 个及以上 `*`**：匹配任意数量的字符（0 到无穷）。

**位置规则：** 通配符在哪一侧，该侧就允许有额外内容；没有通配符的一侧必须是字符串的边界。

| 模式 | 含义 | 匹配 | 不匹配 |
|---|---|---|---|
| `"***开心***"` | 任意位置包含"开心" | "很开心啊"、"开心"、"我很开心" | — |
| `"***开心"` | 以"开心"结尾 | "很开心"、"开心" | "开心了" |
| `"开心***"` | 以"开心"开头 | "开心果"、"开心" | "很开心" |
| `"*开心*"` | 前后各恰好 1 字符 | "很开心啊" | "开心"、"超开心啊呀" |
| `"**民主主义"` | 前面恰好 2 字符 | "社会民主主义" | "民主主义"、"新民主主义" |

组合示例：`["心情" == "***开心***"|"心情" == "***高兴***"]开心时表现` —— 心情变量中包含"开心"或"高兴"时注入。

#### B-3c 存在性检查

检查变量是否已被定义（存在且非 null / undefined）。

| 语法 | 含义 | 示例 |
|---|---|---|
| `["变量名" ?]` | 变量已定义时为真 | `["圣杯" ?]` |
| `["变量名" !?]` | 变量未定义时为真 | `["圣杯" !?]` |

用例：`["圣杯" ?]圣杯持有者的特殊剧情` —— 只有在变量"圣杯"被创建之后才注入这段设定。

#### B-3d 嵌套属性访问

变量为对象类型时，使用 `/` 作为路径分隔符访问深层属性。适用于所有运算符。

```text
["角色/状态/HP" >= 60]
["角色/装备/武器" == "圣剑"]
["世界/天气" == "暴风雨"]
["角色/技能列表" ∋ "***火焰***"]
```

路径中的每一段对应对象的一个层级。若路径中任何一层不存在，条件判定为假（不报错，静默不匹配）。

此处的 `/` 路径格式与姊妹脚本 VarUpdate 中用于定位和操作嵌套属性的路径表达格式相同。两个脚本共用同一套路径规范，确保用户在标签条件和变量更新指令中使用一致的寻址方式。

#### B-3e 数组操作

| 运算符 | 含义 | 语法 | 说明 |
|---|---|---|---|
| `∋` | 数组含值 | `["物品栏" ∋ "圣杯"]` | 按值查找：数组中是否存在等于指定值的元素 |
| `!∋` | 数组不含值 | `["物品栏" !∋ "圣杯"]` | 按值排除 |

值支持通配符：`["物品栏" ∋ "***圣杯"]` —— 物品栏中是否有以"圣杯"结尾的元素。

#### B-3f 对象/字典操作

| 运算符 | 含义 | 语法 | 说明 |
|---|---|---|---|
| `⊇` | 对象含键 | `["属性" ⊇ "力量"]` | 按键名查找：对象中是否存在指定键 |
| `!⊇` | 对象不含键 | `["属性" !⊇ "力量"]` | 按键名排除 |

键名支持通配符：`["物品栏" ⊇ "***元素魔法书"]` —— 物品栏对象中是否有键名以"元素魔法书"结尾的条目。

#### B-3g 集合长度判断

使用 `#` 运算符获取数组或对象的元素数量，再与数值进行比较。

| 语法 | 含义 |
|---|---|
| `["物品栏" # >= 3]` | 物品栏至少有 3 个元素 |
| `["队伍成员" # == 4]` | 队伍恰好 4 人 |
| `["物品栏" # > 0]` | 物品栏非空 |

`#` 后接任意数值比较运算符（`>` `>=` `<` `<=` `==` `!=`）。

#### B-3h 变量与变量比较

比较两个变量的值（而非变量与字面量）。使用 `$"变量名"` 语法标记右侧为变量引用。

| 语法 | 含义 |
|---|---|
| `["当前HP" <= $"最大HP"]` | 当前 HP 的值 ≤ 最大 HP 的值 |
| `["地点" == $"目标地点"]` | 当前地点等于目标地点 |
| `["攻击力" > $"敌人/防御力"]` | 我方攻击力 > 敌人的防御力（支持嵌套路径） |

右侧的 `$"..."` 同样支持 `/` 路径访问嵌套属性。

### B-4 特殊标签

| 标签 | 含义 |
|---|---|
| `[NonPrompts]` | 该条目在任何步骤中都不被发送。无论当前全局策略是「默认包含」还是「默认排除」。 |
| `[All]` | 该条目在所有步骤中都被发送。无论当前全局策略是「默认包含」还是「默认排除」。 |

### B-5 标签与酒馆原生管理策略的关系

- 标签系统是在酒馆原生条目管理之上的**附加过滤层**，而非替代。
- 酒馆原生的控制逻辑依然优先生效：
  - 若一个世界书条目处于**关闭状态**（红灯），无论是否携带标签，该条目都不会被发送。
  - 若一个世界书条目设为**关键词激活**（绿灯），只有当上下文中出现匹配关键词时才会被发送，标签不会绕过这一规则。
  - 若一个世界书条目设为**永久激活**（蓝灯），则只要标签条件满足即发送。
- 判定流程：酒馆原生判定「该发送」→ 标签系统判定「智能体匹配 + 变量条件是否全部满足」→ 最终注入。

### B-6 无标签条目的全局策略

对于未携带任何标签（既没有智能体匹配标签，也没有变量条件标签，也没有特殊标签）的预设/世界书条目，脚本提供两套可切换的全局策略：

- **策略 A「默认包含」（出厂默认）**：未标记的条目在所有智能体步骤中都会被注入。若希望某条目完全不参与任何步骤，需加入 `[NonPrompts]`。
- **策略 B「默认排除」**：未标记的条目在任何步骤中都不会被注入。若希望某条目参与所有步骤，需加入 `[All]`。

### B-7 发送历史楼层数

- 每个智能体可独立配置「发送历史楼层数」，即在构建该步骤的上下文时，包含多少条最近的历史聊天记录。
- 此设置允许不同步骤使用不同长度的上下文窗口。例如：
  - 主生成步骤可能需要较多历史（如 20 条）以保持剧情连贯。
  - 变量更新步骤可能只需最近几条（如 3 条）即可完成解析任务。
  - 摘要/检索步骤可能需要完整历史以总结全局。
- 默认值为沿用酒馆当前设置的上下文长度（即不做额外限制）。

### B-8 完整运算符速查表

| 类别 | 运算符 | 语法示例 | 说明 |
|---|---|---|---|
| 智能体匹配 | 标签名 | `[创意写作]` | 在该步骤中注入 |
| OR（同类型） | `|` | `[A|B]`、`["x"==1|"y"==2]` | 同一 `[]` 内同类型项满足任一 |
| AND | 多个`[]` | `[A]["x"==1]` | 全部满足才注入 |
| NOT | `!` 前缀 | `[!变量更新]`、`[!"HP">=60]` | 对标签/条件取反，通用规则 |
| 宽松相等 | `==` | `["地点"=="酒馆"]` | 不校验类型 |
| 宽松不等 | `!=` | `["地点"!="酒馆"]` | 不校验类型 |
| 严格相等 | `===` | `["HP"===100]` | 校验类型 |
| 严格不等 | `!==` | `["HP"!=="100"]` | 校验类型 |
| 大于 | `>` | `["HP">0]` | number 类型 |
| 大于等于 | `>=` | `["HP">=60]` | number 类型 |
| 小于 | `<` | `["HP"<80]` | number 类型 |
| 小于等于 | `<=` | `["HP"<=100]` | number 类型 |
| 存在性 | `?` / `!?` | `["圣杯"?]` | 变量是否已定义 |
| 通配符 | `*` | `["x"=="***火***"]` | 所有字符串场景可用 |
| 数组含值 | `∋` / `!∋` | `["背包"∋"剑"]` | 按值查找 |
| 对象含键 | `⊇` / `!⊇` | `["属性"⊇"力量"]` | 按键名查找 |
| 集合长度 | `#` + 比较 | `["背包"#>=3]` | 元素数量判断 |
| 嵌套访问 | `/` 路径 | `["角色/HP">=60]` | 深层属性 |
| 变量引用 | `$"var"` | `["HP"<=$"MaxHP"]` | 变量与变量比较 |
| 全局控制 | `[NonPrompts]` / `[All]` | — | 特殊标签 |

### B-9 标签系统对正则规则的管辖

标签系统不仅管辖预设和世界书条目的注入，也管辖酒馆正则规则（Regex Scripts）的应用。用户可在正则规则的名称（script_name）中写入标签（语法与 B-1 完全一致），控制该正则在哪些智能体步骤中生效。

#### B-9a 标签写入方式

在酒馆的正则管理窗口中，将标签写在正则规则的名称字段中。例如：

- `格式化对话 [创意写作]` — 仅在「创意写作」步骤中应用此正则。
- `清理OOC标记 [!变量更新]` — 除「变量更新」步骤外均应用。
- `统一标点 [All]` — 所有步骤中均应用。

标签语法（智能体匹配、变量条件、特殊标签、逻辑组合）与 B-1 ~ B-4 中定义的完全一致，复用同一套解析规则。未携带任何标签的正则规则遵循 B-6 定义的全局策略（默认包含或默认排除）。

#### B-9b 按实际处理情境决定标签介入

酒馆的正则系统在多种处理情境下被调用。脚本根据**当前实际正在执行的处理情境**来判断标签系统是否应当参与过滤——而非根据正则规则自身声明的作用范围设置来判断。

酒馆正则的处理情境由两个维度组成：

- **消息来源**（source）：用户输入 / AI 输出 / 快捷命令 / 世界信息 / 推理
- **处理目标**（destination）：真实替换（两个格式选项均未勾选）/ 仅格式显示 / 仅格式提示词 / 两者均勾选

标签系统的介入规则：

| 当前实际情境 | 标签系统是否介入 | 原因 |
|---|---|---|
| 消息来源 = AI 输出，处理目标 = 真实替换 | ✅ 介入 | AI 输出来自某个具体智能体步骤 |
| 消息来源 = 推理，处理目标 = 真实替换 | ✅ 介入 | 推理内容同样来自某个智能体步骤 |
| 消息来源 = 任何，处理目标 = 仅格式提示词（或同时勾选了格式提示词） | ✅ 介入 | 正在为某个智能体步骤组装提示词 |
| 消息来源 = 用户输入，处理目标 = 任何 | ❌ 不介入 | 用户输入不属于任何智能体步骤 |
| 消息来源 = 快捷命令，处理目标 = 任何 | ❌ 不介入 | 快捷命令与智能体步骤无关 |
| 消息来源 = 任何，处理目标 = 仅格式显示 | ❌ 不介入 | 纯视觉层呈现，不影响智能体执行 |

当同一条正则规则同时声明了标签需要介入和不需要介入的作用范围（例如消息来源同时勾选了「AI 输出」和「用户输入」）时，不需要特殊处理。每次调用正则引擎时有一个确定的实际处理情境，脚本据此判断是否以标签过滤——同一条正则规则在不同情境下分别走不同的判断路径。

#### B-9c 「被编辑时应用」的处理

酒馆正则的「被编辑时应用」（run_on_edit）选项控制正则是否在消息被手动编辑时重新触发。在多智能体管道中，后续步骤对消息的写入操作（追加、覆写）也可能被酒馆视为"编辑"，从而触发 `run_on_edit` 正则——此时无法确定应以哪个智能体步骤的标签上下文来过滤正则规则。

为消除此类歧义，脚本对携带智能体特定标签的正则规则**强制禁用** `run_on_edit`：

- **智能体特定标签**（如 `[创意写作]`、`[!变量更新]`）：`run_on_edit` 被强制置为关闭状态。在酒馆的正则编辑界面中，该选项显示为未勾选、灰色不可编辑，并附带说明文字表明此限制由标签系统施加。
- **全步骤生效标签**（如 `[All]`）或**未携带任何标签的正则**：`run_on_edit` 保持用户设置不变，脚本不干预。

## 三、后处理策略

后处理策略决定每个智能体步骤的生成结果如何处理和写入聊天记录。每个智能体独立配置自己的后处理策略。

### C-1 写入方式（五选一）

| 选项 | 行为 |
|---|---|
| **仅解析** | 生成结果**不写入**聊天记录（不出现在对话上下文中），仅通过事件广播供其他脚本（如 VarUpdate）消费。 |
| **独立楼层 (User)** | 生成结果作为一条新的 **User 角色**消息插入聊天记录。 |
| **独立楼层 (Assistant)** | 生成结果作为一条新的 **Assistant 角色**消息插入聊天记录。 |
| **追加到最新楼层末尾** | 将生成结果拼接到当前最新一条消息的末尾（无论该消息属于 User 还是 Assistant）。 |
| **覆写最新楼层** | 用生成结果**完全替换**当前最新一条消息的内容（无论该消息属于 User 还是 Assistant）。 |

**关于追加和覆写的作用对象**：这两种方式操作的是「当前聊天记录中最新的一条消息」，不区分该消息的角色。这意味着：

- 若流水线中排序第一的智能体（A）选择「覆写最新楼层」，它会覆写用户刚发送的输入消息。用例：用户仅输入简短指令（如"前往集市"），由 AI 根据指令和设定生成详细的角色行为描写，替换掉原始指令，实现 AIRPG（AI-driven RPG，即由 AI 驱动的角色扮演游戏）式交互。
- 若排序第一的智能体（A）选择「追加到最新楼层末尾」，它的输出会追加到用户消息之后。用例：先用一个快速模型扫描上下文、列出关键词以激活相关世界书条目，为后续推理模型预筛选提示词，提高 Token 效率和注意力效价。

**关于事件广播**：无论选择哪种写入方式，脚本都会在每个步骤完成后广播一个**消息完成事件**。事件携带该步骤生成的**完整最终消息内容**（即经过全部预处理后的内容），供其他脚本（如 VarUpdate）消费。对于写入聊天记录的方式，脚本同时触发 SillyTavern 的标准消息事件（如 `CHARACTER_MESSAGE_RENDERED`），确保所有监听标准事件的脚本也能正常工作。「仅解析」的含义仅仅是不将结果写入对话上下文，但消息完成事件照常广播。

若启用了调试模式（见 G-1），每个步骤完成时还会将完整的最终消息内容输出到浏览器控制台（Console），便于开发者检查实际发送和接收的内容。

> **与 VarUpdate 的配合**：消息完成事件在所有写入方式下都会广播，包括「仅解析」模式。这意味着即使生成结果不写入聊天记录，VarUpdate 仍可通过监听此事件获取消息内容，识别其中的变量标签并执行解析。这是「仅解析」模式的典型用例之一：让智能体生成变量更新指令，不写入对话但仍触发变量更新。

### C-2 后处理选项

以下选项对所有写入方式生效，用于在将结果写入（或广播）之前对原始输出进行后处理。多个选项同时启用时，按以下固定顺序依次执行：

1. 推理过程剥离（由 A-5a「传回推理过程」控制：不勾选时自动剥离）
2. 裁剪开始标签前（C-2a）
3. 裁剪结束标签后（C-2b）

#### C-2a 裁剪开始标签前

- **默认关闭。**
- 开启后，用户可自定义一个「开始标签」（如 `<正文>`、`<output>` 等任意字符串）。
- 写入前，脚本在原始输出中查找该标签的**首次出现位置**，删除该标签及其之前的所有内容，仅保留标签之后的部分。
- 若输出中未找到该标签，则不做裁剪，保留完整输出。

#### C-2b 裁剪结束标签后

- **默认关闭。**
- 开启后，用户可自定义一个「结束标签」（如 `</正文>`、`</output>` 等任意字符串）。
- 写入前，脚本在原始输出中查找该标签的**最后一次出现位置**，删除该标签及其之后的所有内容，仅保留标签之前的部分。
- 若输出中未找到该标签，则不做裁剪，保留完整输出。

> 两者可同时启用。同时启用时，效果为：查找开始标签首次出现的位置和结束标签最后一次出现的位置，提取两者之间的内容作为最终写入结果。

#### C-2c 拼接前换行（仅「追加到最新楼层末尾」时显示）

- 仅当写入方式选择「追加到最新楼层末尾」时，此选项才会显示。
- **默认勾选。**
- 勾选时：在原消息内容和追加内容之间插入一个换行符。
- 取消时：直接拼接，不插入换行符。

## 四、重试策略

当某个智能体步骤的生成结果不符合预期时，脚本可自动重试该步骤。每个智能体独立配置重试规则。

### D-1 自定义结束标识符

- 用户可为每个智能体定义一个「结束标识符」字符串（如 `</正文>`、`[END]`、`</output>` 等）。
- 每次生成完成后，脚本检查输出中是否包含该标识符。若不包含，判定为输出不完整，触发重试。
- 此选项**默认关闭**。开启后必须填写标识符字符串。

### D-2 变量解析联动重试

- 用户可为每个智能体开启此选项。
- 开启后，该步骤生成完成并广播消息完成事件后，脚本进入**等待状态**，监听 VarUpdate 的事件反馈：
  - 收到 `变量更新完成` → 判定成功，继续执行下一步骤。
  - 收到 `变量更新失败` → 判定失败，广播 `步骤重试` 事件（通知 VarUpdate 回退变量），然后重新生成。
  - 超时未收到任何事件 → 按超时处理（见 D-4 单次生成超时），计入重试次数。
- VarUpdate 根据其容错阈值判定丢弃指令数是否可接受：少量指令丢弃（≤ 阈值）仍视为成功广播 `变量更新完成`，超过阈值或整段解析失败才广播 `变量更新失败`。
- 此机制通过事件通信实现松耦合：本脚本仅广播消息内容并监听结果事件，不关心变量解析的具体规则；VarUpdate 仅关心消息中是否有变量标签，不关心生成流程。
- 此选项**默认关闭**。

### D-3 重试策略叠加

- D-1（结束标识符检查）和 D-2（变量解析联动）可同时启用。
- 同时启用时的执行顺序为：先检查结束标识符是否存在 → 通过后再等待 VarUpdate 确认解析结果 → 任一环节判定失败即触发重试。

### D-4 重试参数

| 参数 | 说明 | 默认值 |
|---|---|---|
| **最大重试次数** | 单个步骤允许的最大重试次数。超过后暂停整条流水线，向用户弹出提示。 | 3 次 |
| **重试间隔** | 两次重试之间的等待时间（秒）。 | 1 秒 |
| **单次生成超时** | 单次生成请求的最大等待时间（秒）。超过后视为该次生成失败，计入重试次数并触发下一次重试。 | 300 秒 |

## 五、执行流程与状态指示

### E-1 按序执行

- 用户发送消息（或点击生成按钮）后，脚本按智能体字母排序（A → B → C → ……）依次执行每个启用的步骤（禁用的步骤自动跳过）。
- 每个步骤完成全部流程（生成 → 重试判定 → 后处理 → 写入/广播）后，才进入下一个步骤。
- 整条流水线执行完毕后恢复空闲状态。

### E-2 自定义按钮（替代酒馆原生底部栏按钮）

脚本加载后，酒馆原生底部栏的「发送」和「中止」按钮始终保持隐藏状态，全程由脚本自定义按钮替代。无论空闲还是执行中，用户看到的都是脚本提供的按钮。脚本卸载或禁用后，原生按钮自动恢复。

#### 空闲状态（未执行流水线时）

- 显示自定义的「发送」按钮，外观与酒馆原生发送按钮一致（可使用相同图标），位置也相同。
- 点击后触发脚本的流水线启动逻辑（而非酒馆原生的单次生成）。

#### 执行中状态（流水线正在运行时）

- 按钮变为一个**旋转的圆形动画**，圆心内显示当前正在执行的步骤对应的**大写英文字母**（A、B、C……）。
- 用户可直观看到当前正在执行哪个智能体步骤。随着步骤切换，圆心字母随之更新。
- 点击该旋转按钮等同于「中止」：立即停止当前步骤的生成请求，终止整条流水线。

#### 流水线结束或中止后

- 按钮恢复为空闲状态的「发送」按钮外观。

### E-3 用户手动中止

- 用户可在流水线执行的任何阶段通过点击状态按钮（E-2）中止执行。
- 中止后：当前步骤已有的输出按其后处理策略处理（若已部分生成则保留已有部分），后续步骤不再执行。
- 用户可通过酒馆原生的消息管理功能（删除消息、撤回等）自行清理不需要的输出。

## 六、配置持久化与分享

### F-1 存储模型

脚本的配置由三层对象组成，各自独立存储：

| 对象 | 内容 | 存储方式 |
|------|------|----------|
| **智能体** | 单个智能体的全部配置（J-1） | 独立 JSON，可单独导入/导出/分享 |
| **方案** | 一组智能体的引用列表及其排序和激活状态 | 独立 JSON，可单独导入/导出/分享 |
| **全局设置** | 脚本级配置（J-2） | global 层持久化 |

- **智能体是独立实体**，不附属于任何方案。同一个智能体可被多个方案引用。
- **方案是引用列表**，记录「哪些智能体参与、排在什么位置、是否激活」，不复制智能体配置本身。
- 切换方案时：方案中引用但本地不存在的智能体 → 自动创建并激活；本地存在但方案未引用的智能体 → 移至待命区（不删除）。

### F-2 导入与导出

#### 智能体级

- 单个智能体可导出为 JSON 文件（含全部配置）。
- 导出时可选择是否包含 API Key（不包含便于安全分享）。
- 导入时若检测到格式不兼容，给出明确提示。

#### 方案级

- 方案可导出为 JSON 文件（含引用的智能体列表和排序）。
- 方案 JSON 可选择**仅引用**（仅导出智能体名称列表，对方需已有同名智能体）或**完整打包**（嵌入所有引用智能体的完整配置，便于整套分享）。
- 导入完整打包的方案时，已存在的同名智能体不覆盖（保留本地版本），不存在的智能体自动创建。

### F-3 内置资源

- 脚本预置若干**内置方案**和**内置智能体**，供新用户快速上手。
- 内置资源以只读形态展示。用户若需修改，可复制一份后编辑副本。

## 七、通知与日志

### G-1 两类信息源

脚本的信息输出分为两类，遵循不同的可见性规则：

**用户主动触发的操作反馈**（连接测试、试生成、Dry Run）：用户点击按钮后等待反馈，结果**始终弹出 toastr**（silence 等级除外），不受通知等级限制。成功弹绿色，失败弹红色。详细内容同时输出到控制台。

**流水线自动事件**（步骤完成、重试、超时等）：由通知等级控制可见性（见 G-2）。

| 信息 | toastr | 控制台 |
|------|--------|--------|
| **用户操作反馈** | | |
| 连接测试成功 / 失败 | ✅ 始终弹（🟢/🔴） | 🟢/🔴 始终输出 |
| 试生成成功 / 失败 | ✅ 始终弹（🟢/🔴，摘要） | 🟢/🔴 状态 + 🔵 完整文本 |
| Dry Run 完成 | ✅ 始终弹（🟢，显示总 Token 数） | 🔵 完整 prompt 列表 |
| **流水线自动事件** | | |
| 步骤完成（耗时 + 字符数） | always 级弹 🟢 | 🟢 always 级输出 |
| 步骤重试 | notice 级弹 🟠 | 🟠 notice 级输出 |
| API 配置 / 标签过滤 / 预处理细节 | 不弹 | 🔵 仅 debug 输出 |
| 步骤彻底失败 / 流水线中止 | error 级弹 🔴 | 🔴 error 级输出 |

### G-2 通知等级

通知等级仅控制**流水线自动事件**的可见性：

| 等级 | toastr 弹窗 | 控制台输出 |
|------|-------------|-----------|
| `debug` | 全部显示 | 全部输出（含蓝色调试信息） |
| `always` | 通过+警告+错误 | 通过+警告+错误 |
| `notice` | 警告+错误 | 警告+错误 |
| `error` | 仅错误 | 仅错误 |
| `silence` | 不显示 | 不输出 |

- **debug**：开发排错。输出每步的 API 请求配置、标签过滤结果、后处理策略执行细节、重试判定过程。
- **always**：日常使用。每个步骤完成成功或失败时提示。
- **notice**（默认）：仅在出现问题时通知（重试触发、超时、失败）。
- **error**：最少打扰。仅报告失败（API 调用失败、重试耗尽）。
- **silence**：完全静默。用户操作反馈也不弹窗。

### G-3 控制台输出格式

所有控制台输出使用统一前缀 `[Agents]`，按四色分级：

| 颜色 | 含义 | 示例 |
|------|------|------|
| 🔵 蓝色 | 调试信息（仅 debug 等级可见） | `[Agents] 步骤 B 标签过滤: 12 条注入, 5 条屏蔽` |
| 🟢 绿色 | 正常通过 | `[Agents] 步骤 A 完成 (2.3s, 1280 chars)` |
| 🟠 橙色 | 警告信息 | `[Agents] 步骤 B 重试 (2/3): 未找到结束标识符` |
| 🔴 红色 | 错误信息 | `[Agents] 步骤 C 失败: API 超时 (300s)` |

## 八、辅助工具

### H-1 提示词预览（Dry Run）

- 每个智能体配置界面提供一个「预览提示词」按钮。
- 点击后，脚本执行该步骤的完整提示词组装流程（标签过滤、历史楼层截取、世界书注入等），但**不实际发送 API 请求**。
- 结果通过通知系统反馈：toastr 弹窗显示总预估 Token 数，控制台输出完整的 prompt 列表（按发送顺序排列的每条 message，含角色、内容和预估 Token 数）。
- 用于调试提示词配置：确认标签过滤是否正确、上下文长度是否合理、是否遗漏或多余了条目。

### H-2 连接测试

- 每个智能体配置界面提供「快速测试」按钮。
- 向目标 API 地址发送一个轻量请求（如模型列表请求），验证地址可达性和 API Key 有效性。
- 结果通过通知系统反馈：成功弹绿色 toastr，失败弹红色 toastr 并显示具体错误（网络不可达、401 未授权、超时等）。

### H-3 试生成

- 每个智能体配置界面提供「试生成」按钮。
- 使用当前智能体的完整配置（API 地址、Key、模型、参数），发送一次真实的生成请求。
- 请求使用一段**脚本内部管理的测试提示词**。该提示词由用户在脚本设置中自由配置（提供合理的默认值），所有智能体共用一份默认值，也可在单个智能体的配置中单独覆盖。
- 结果通过通知系统反馈：成功弹绿色 toastr（摘要），失败弹红色 toastr（错误原因）。完整的生成文本输出到控制台蓝色日志。
- 试生成能验证完整链路：连接可达 → 认证通过 → 模型名正确 → 参数兼容 → 模型实际返回内容。

## 九、UI 面板

脚本的独立 UI 面板自上而下分为四个区域。

### I-1 工具栏（顶部第一排）

从左到右排列以下按钮：

| 内置方案 | 弹出浮窗，展示预置方案列表（名称 + 简介），可选择加载 |
| 导入新智能体 | 从 JSON 文件导入一个智能体 |
| 创建新智能体 | 创建一个空白智能体 |
| 内置智能体 | 弹出浮窗，展示预置智能体列表（名称 + 简介），可选择加载 |
| 批量编辑模式 | 进入批量选择状态，可多选智能体后批量导出/删除/移动 |

### I-2 方案栏（顶部第二排）

- **左侧**：当前使用的方案名称。点击展开下拉列表，可切换到其他方案。
- **右侧**：紧跟四个操作按钮：

| 按钮 | 操作 |
|------|------|
| 保存方案更新 | 将当前的智能体排序和激活状态写回方案 |
| 重应用当前方案 | 丢弃未保存的排序/激活调整，恢复到方案记录的状态 |
| 导出当前方案 | 将当前方案导出为 JSON（见 F-2） |
| 删除当前方案 | 删除当前方案（智能体本身不受影响） |

### I-3 智能体排序与激活区（中间主体）

中间区域分为**待命区（上方）**和**工作区（下方）**两个区域，通过在两区之间移动智能体来表达启用/禁用状态，而非传统的开关按钮。

- **工作区**：其中的智能体按从上到下的顺序参与流水线执行（对应字母标识 A → B → C → ……）。
- **待命区**：其中的智能体在流水线执行时被跳过，配置保留不丢失。
- 待命区的智能体以**灰暗、半透明**的样式显示，与工作区形成视觉区分。

#### 智能体条目

每个智能体显示为一个长条，布局如下：

```text
[⠿ 拖拽] 智能体名称                    [📝][✏️][📋] [⇅]
   左对齐                                    右对齐
```

| 位置 | 元素 | 操作 |
|------|------|------|
| 左侧 | 拖拽手柄 | 拖动调整排序（工作区内、待命区内、跨区拖动均可） |
| 左侧 | 名称 | 显示智能体名称 |
| 右侧 | 查看备注 | 弹出气泡显示该智能体的备注文本 |
| 右侧 | 编辑智能体 | 打开智能体配置浮窗（见 I-5） |
| 右侧 | 克隆智能体 | 复制一份新智能体（自动命名为「原名 - 副本」） |
| 右侧 | 切换选区 | 将智能体移动到另一区域的末尾（待命 ↔ 工作） |

#### 方案切换行为

- 切换方案时，方案中引用但本地不存在的智能体 → 自动创建并放入工作区。
- 本地存在但新方案未引用的智能体 → 全部移至待命区（不删除）。
- 方案中引用且本地已存在的智能体 → 按方案记录的排序放入工作区。

### I-4 全局设置栏（底部）

底部区域左右分栏：

- **左侧**（上下排列两个下拉选择器）：
  - 通知等级：debug / always / notice / error / silence（见 G-2）
  - 排除策略：默认包含 / 默认排除（见 B-6）
- **右侧**：试生成提示词的富文本编辑器（见 H-3）
