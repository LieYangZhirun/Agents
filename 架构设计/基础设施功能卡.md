# 多智能体脚本（Agents）——基础设施功能卡

> 本文档定义 Agents 脚本的底层基础设施——支撑面向用户功能卡中所有用户可见功能所需的技术组件、设计决策和工程化方案。

## 〇、酒馆原生管道分析与接入策略

> 本节记录 SillyTavern 从用户发送消息到 AI 回复渲染完成的完整提示词拼装管道。Agents 脚本的核心设计决策——**管道镜像 + 标签过滤层**——基于此分析得出。

### 〇-2. 正则在管道中的五个作用点

酒馆的正则系统并非在某个统一环节一次性执行，而是分散在管道的五个不同位置，各有不同的语义：

| 序号 | 管道位置 | 调用参数 | 语义 | 标签系统是否需要介入 |
|------|----------|----------|------|---------------------|
| ③ | 用户输入 | `placement=USER_INPUT` | 真实替换用户消息 | 否 — 来源非 AI，不属于标签管辖 |
| ④ | 聊天历史→提示词 | `placement=USER_INPUT/AI_OUTPUT, isPrompt=true` | 仅格式提示词 | 部分 — 仅 AI_OUTPUT 部分需标签过滤，USER_INPUT 部分不需要 |
| ⑤ | 世界书条目 | `placement=WORLD_INFO, isPrompt=true` | 仅格式提示词 | 是 — 面向 AI 的情境 |
| ⑨ | AI回复 | `placement=AI_OUTPUT` | 真实替换 | 是 — 来源为 AI 输出 |
| ⑪ | 消息渲染 | `isMarkdown=true` | 仅可视化效果 | 否 — 面向用户的显示 |

> 此表与面向用户功能卡 B-9b 的设计一致。标签过滤必须在上述每个"是"的环节独立执行，不可合并。

### 〇-3. 三个原生系统的数据访问方式

#### 预设（Preset）

- **数据来源**：`promptManager.getPromptCollection(type)` 返回有序 Prompt 集合
- **标签位置**：每个 Prompt 的 `name` 字段（即预设条目标题）
- **原生启用判断**：`promptManager.isPromptDisabledForActiveCharacter(identifier)`
- **特点**：
  - 预设条目不存在类似世界书的激活状态（红/绿/蓝灯），只有简单的启用/禁用
  - 预设定义了提示词拼装的**结构**（槽位顺序），世界书和聊天历史是往结构里**填充内容**
  - 预设条目的 `content` 已经是最终文本，不需要额外的激活判定

#### 世界书（World Info）

- **数据来源**：酒馆原生扫描函数产出的 `allActivatedEntries`（Map，值为完整条目对象）
- **标签位置**：每个条目的 `comment` 字段（即世界书条目备注/标题）
- **原生激活判定**：由酒馆本体执行（constant / 关键词匹配 / secondary / 递归 / 概率 / inclusion group / sticky / cooldown / delay 等），逻辑极为复杂
- **关键约束**：
  - 世界书条目在被激活后、合并为字符串前的窗口期（步骤 5b-5c）内仍然是独立对象
  - `WORLDINFO_SCAN_DONE` 事件（步骤 5d）的回调参数中可以访问完整条目对象
  - 一旦合并为字符串（步骤 5c），条目粒度信息不可逆地丢失

#### 正则（Regex Scripts）

- **数据来源**：`getRegexScripts({ allowedOnly: true })` 返回全部生效规则
- **标签位置**：每条规则的 `scriptName` 字段（即正则脚本名称）
- **正则来源层级**：全局（GLOBAL）→ 角色卡（SCOPED）→ 预设（PRESET），按优先级排列
- **关键属性**：
  - `placement[]`：适用的消息来源类型
  - `markdownOnly` / `promptOnly`：情境过滤标志
  - `runOnEdit`：编辑消息时是否重新应用（带标签的规则强制禁用此项）
  - `disabled`：原生禁用标志

### 〇-4. 核心设计决策：管道镜像 + 标签过滤层

#### 决策背景

Agents 脚本需要为每个 Agent 步骤构建不同的提示词（不同的预设条目、世界书条目、正则规则组合），因此不能直接复用酒馆原生的 `Generate()` 流程。

两个候选策略：

| 维度 | 策略A：管道镜像 | 策略B：事件钩子拦截 |
|------|----------------|-------------------|
| 方法 | 镜像酒馆管道结构，在每个环节加标签过滤 | 监听酒馆事件，在回调中修改数据 |
| 适用场景 | 多步骤管道（每步需不同配置） | 单次拦截（全局统一过滤） |
| 行为一致性 | 高（结构相同，仅多一层过滤） | 中（依赖酒馆内部实现细节） |
| 维护成本 | 需同步酒馆管道变更 | 需追踪酒馆事件契约变更 |
| 代码量 | 较多（镜像管道代码） | 较少（但 hack 性质重） |

#### 最终决策：采用策略A（管道镜像）

**理由**：

1. **多 Agent 管道的本质需求**：每个 Agent 步骤需要不同的预设/世界书/正则组合，这要求完全独立的提示词构建流程，不是单次拦截能解决的。

2. **行为一致性保障**：管道结构与酒馆原生保持一致，用户在酒馆原生发信和通过 Agents 发信时的行为差异最小化。偏离原生管道结构会导致隐蔽的行为不一致 bug，排查极其困难。

3. **代码重量的 trade-off**：增加的代码大部分是对酒馆已有函数的**调用与编排**（而非重新实现），集中在"搬运逻辑"而非"创造逻辑"。每一步都可对照酒馆源码验证正确性。

#### 世界书激活判定策略

酒馆原生的世界书激活判定逻辑极为复杂（关键词匹配、secondary 关键词、递归扫描、概率、inclusion group、sticky/cooldown/delay 等）。

**决策：复用酒馆原生扫描，在激活结果上做标签过滤。** 不重新实现世界书激活逻辑。

具体实现：通过 `WORLDINFO_SCAN_DONE` 事件获取激活后的完整条目对象集合，按标签过滤后再进行合并。

### 〇-5. Agents 管道镜像流程

以下流程定义每个 Agent 步骤的执行过程。管道结构镜像酒馆原生流程（〇-1），唯一差异是在涉及预设/世界书/正则的环节各加一层标签过滤。

```
对于每个 Agent 步骤（以当前 Agent 的标签集合为上下文）:
│
├─ ① 拦截原生 Generate，启动 Agents 管道
│
├─ ② 用户消息处理（仅首步，与原生一致）
│   → 应用 USER_INPUT 正则（无需标签过滤，来源非 AI）
│
├─ ③ 构建聊天历史
│   遍历 coreChat（过滤掉系统消息后的聊天历史），对每条消息:
│   → 读取正则规则 → 解析 scriptName 中的标签 → 按当前 Agent 过滤
│   → 用过滤后的正则集执行 getRegexedString(... {isPrompt:true})
│   ※ 与原生 script.js L4337-4365 逻辑一致
│
├─ ④ 世界书扫描与合并
│   → 调用酒馆原生扫描函数，获取 allActivatedEntries
│   → 对激活后的条目：读 entry.comment → parseLabels() → 按当前 Agent 过滤
│   → 被保留的条目经 WORLD_INFO 正则处理（正则集同样经标签过滤）
│   → 按 position 分组合并为 worldInfoBefore / worldInfoAfter 字符串
│   ※ 与原生 world-info.js L4997-5059 逻辑一致
│
├─ ⑤ 预设拼装
│   → promptManager.getPromptCollection(type) 获取有序条目集合
│   → 对每个条目:
│     ├─ 酒馆原生 enabled/disabled 判断（优先）
│     ├─ 读 prompt.name → parseLabels() → 按当前 Agent 过滤
│     └─ 保留通过的条目
│   → 世界书字符串填入对应槽位（worldInfoBefore / worldInfoAfter）
│   → 聊天历史填入对应槽位
│   → 按 token 预算裁剪
│   → 产出最终消息数组
│   ※ 与原生 openai.js L1078-1409 逻辑一致
│
├─ ⑥ API 调用
│   → 使用酒馆助手的 generateRaw() 或直接 sendOpenAIRequest()
│
├─ ⑦ AI 回复处理
│   → 读取正则规则 → 解析标签 → 按当前 Agent 过滤
│   → 用过滤后正则集执行 AI_OUTPUT 真实替换
│   → 用过滤后正则集执行 REASONING 真实替换
│   → 回复写入 chat[]
│   ※ 与原生 script.js L6260 / L5332 逻辑一致
│
├─ ⑧ 决策：触发下一个 Agent 步骤 or 结束管道
│   → 中间步骤：将回复作为下一步骤的上下文输入
│   → 最终步骤：触发 addOneMessage 渲染
│   → 渲染环节的显示正则由酒馆原生处理（标签系统不干预 isMarkdown 情境）
│
└─ （管道结束）
```

**核心原则**：管道结构和步骤顺序与酒馆原生保持一致，唯一差异是每个涉及预设/世界书/正则的环节都多一层标签过滤。以此最大程度减少因管道行为差异导致的隐蔽 bug。

核心调用链路（一次完整的多 Agent 生成）：

```
主控制器.startPipeline()
  → 遍历启用的 Agent（按字母排序）
    → 模块6: 读取当前 Agent 配置
    → 模块2: 组装提示词（内部调用模块1过滤预设/世界书/正则）
    → 模块3: 发送 API 请求
    → 模块4: 后处理（裁剪、写入）
    → 模块5: 重试判定（可能调用模块7与VarUpdate通信）
    → 模块7: 广播步骤完成事件
    → 模块8: 输出通知
  → 全部完成 → 广播管道结束事件
```

## 三、各模块详细规格

### 模块 1：标签解析引擎

> 支撑功能：B-1 ~ B-9（标签语法、智能体匹配、变量条件、特殊标签、正则管辖）

#### 职责

接收一段文本（预设标题 / 世界书 comment / 正则 script_name），解析其中的 `[标签]` 语法，结合当前 Agent 的上下文（标签名、变量状态），输出**是否注入**的布尔判定。

#### 接口

```typescript
interface LabelContext {
  /** 当前 Agent 的标签名 */
  agentLabel: string;
  /** 全局策略：'include' | 'exclude' */
  globalPolicy: 'include' | 'exclude';
  /** 变量读取函数（读取 message 层 data 字段） */
  readVariable: (path: string) => any;
}

/**
 * 判定一个条目是否应在当前 Agent 步骤中注入
 * @param text 包含标签的文本（预设标题 / 世界书 comment / 正则 script_name）
 * @param ctx 当前 Agent 的上下文
 * @returns true = 注入, false = 过滤掉
 */
function shouldInject(text: string, ctx: LabelContext): boolean;
```

#### 判定流程

```text
1. 调用公共库 label-parser 解析所有 [...] 标签
2. 若无任何标签 → 按 globalPolicy 决定（include → true, exclude → false）
3. 检查特殊标签：
   ├─ [NonPrompts] → return false
   └─ [All] → return true
4. 遍历所有标签，按 AND 逻辑组合：
   ├─ 智能体匹配标签 → 检查 agentLabel 是否匹配（含 OR/NOT）
   └─ 变量条件标签 → 调用 readVariable 读取值，执行比较运算
5. 所有标签条件都满足 → return true
   任一标签条件不满足 → return false
```

#### 变量条件求值

支持的运算符及求值逻辑与面向用户功能卡 B-3 / B-8 完全对应：

| 运算符 | 求值逻辑 |
|--------|---------|
| `==` / `!=` | 宽松比较（`==`），支持类型自动转换 |
| `===` / `!==` | 严格比较，类型和值均需匹配 |
| `>` `>=` `<` `<=` | 数值比较，非数值时返回 false |
| `?` / `!?` | `readVariable(path) !== undefined && readVariable(path) !== null` |
| `∋` / `!∋` | `Array.isArray(val) && val.some(item => matchValue(item, target))` |
| `⊇` / `!⊇` | `typeof val === 'object' && Object.keys(val).some(k => matchValue(k, target))` |
| `#` + 比较 | `(Array.isArray(val) ? val.length : Object.keys(val).length) op number` |

字符串比较时自动调用公共库 `wildcard-match` 处理 `*` 通配符模式。

变量引用（`$"变量名"`）：先通过 `readVariable` 读取右侧变量的值，再执行比较。

### 模块 2：管道引擎

> 支撑功能：〇-5（管道镜像策略）、B-5（标签与原生管理的关系）、B-7（历史楼层数）

#### 职责

为指定 Agent 步骤组装完整的提示词消息数组。内部镜像酒馆原生管道结构，在每个环节插入标签过滤层。

#### 接口

```typescript
interface PipelineInput {
  /** 当前 Agent 的配置 */
  agentConfig: AgentConfig;
  /** 标签上下文 */
  labelCtx: LabelContext;
}

interface PipelineOutput {
  /** 最终的消息数组（按发送顺序） */
  messages: ChatMessage[];
  /** 预估 Token 数 */
  estimatedTokens: number;
  /** 调试信息（标签过滤统计等） */
  debug: PipelineDebugInfo;
}

/**
 * 为指定 Agent 执行完整的提示词组装管道
 */
function assemblePipeline(input: PipelineInput): Promise<PipelineOutput>;
```

#### 子模块

##### 2a. 预设适配层 (`preset-adapter.ts`)

- 调用酒馆本体的 `promptManager.getPromptCollection()` 获取预设条目有序集合。
- 遍历每个条目，对其 `name`（标题）调用模块 1 的 `shouldInject()` 过滤。
- 被过滤的条目标记为 `disabled`，后续拼装环节自动跳过。
- 酒馆原生的 `enabled` / `isPromptDisabledForActiveCharacter` 判定优先于标签过滤（先由原生判定"该发送"，再由标签判定"是否匹配"）。

##### 2b. 世界书适配层 (`worldinfo-adapter.ts`)

- 监听酒馆事件 `WORLDINFO_SCAN_DONE`，在回调中拦截 `args.activated.entries`。
- 遍历每个激活的世界书条目，对其 `comment` 字段调用模块 1 的 `shouldInject()` 过滤。
- 将不匹配的条目从激活列表中移除（在它们被合并为字符串之前）。
- 世界书原生的激活逻辑（关键词匹配、permanentActivation 等）保持不变——标签是附加过滤层。

##### 2c. 正则适配层 (`regex-adapter.ts`)

- 读取酒馆的正则规则列表（`getRegexScripts()`）。
- 对每条规则的 `script_name` 调用模块 1 的 `shouldInject()` 过滤。
- **按当前实际处理情境决定是否介入**（见 B-9b）：
  - AI 输出真实替换 → 介入
  - 推理真实替换 → 介入
  - 仅格式提示词 → 介入
  - 用户输入 / 快捷命令 / 仅格式显示 → 不介入（使用原始正则集）
- 携带 Agent 特定标签的正则强制禁用 `run_on_edit`（见 B-9c）。

##### 2d. 聊天历史构建器 (`chat-builder.ts`)

- 从 `chat[]`（`coreChat`，排除系统消息）中按 Agent 配置的「历史消息层数」截取最近 N 条消息。
- 对每条消息应用提示词正则（`isPrompt: true`），使用经 2c 过滤后的正则集。
- 产出的消息数组仅用于此次提示词组装，不修改 `chat[]` 原始数据。

### 模块 3：API 调用层

> 支撑功能：A-2 ~ A-5（API 格式、连接参数、模型参数）

#### 职责

根据 Agent 配置构建 API 请求体，发送请求，处理流式/非流式响应。

#### 接口

```typescript
interface ApiRequest {
  /** API 地址 */
  url: string;
  /** API Key */
  apiKey: string;
  /** 模型名称 */
  model: string;

  stream: boolean;

  content: string;
  /** 推理/思维链内容（若有） */
  reasoning?: string;
  /** 实际消耗的 Token 数 (usage) */
  usage?: { prompt_tokens: number; completion_tokens: number };
}

/**
 * 发送 API 请求
 * @param request 请求配置
 * @param onChunk 流式模式下的 chunk 回调
 * @returns 完整响应
 */
function sendRequest(
  request: ApiRequest,
  onChunk?: (chunk: string) => void
): Promise<ApiResponse>;

/**
 * 获取模型列表
 */
function fetchModelList(url: string, apiKey: string, format: string): Promise<string[]>;
```

#### API 参数映射表

存储于 `data/api-params-map.json`，结构：

```json
{
  "openai": {
    "common": ["stream", "max_tokens", "temperature", "top_p"],
    "niche": ["frequency_penalty", "presence_penalty"],
    "defaults": { "temperature": 1.0, "max_tokens": 4096 }
  },
  "claude": {
    "common": ["stream", "max_tokens", "temperature", "top_p"],
    "niche": ["top_k"],
    "defaults": { "temperature": 1.0, "max_tokens": 4096 }
  }
}
```

此映射表驱动 UI 面板的参数显示/隐藏（模块 9 读取），以及请求体构建时的参数筛选（本模块读取）。更新映射表不需要改代码。

**未知格式兜底**：若用户选择的 API 格式在映射表中不存在（如酒馆新版本新增了格式），按「自定义」处理——显示全部参数。在 `debug` 通知等级下，向控制台和 toastr 输出警告：`[Agents] 未知 API 格式 "xxx"，按自定义格式处理`。

#### 请求体构建

根据 `format` 将 `messages` + `params` 转换为对应 API 格式的请求体。各格式的请求体差异（如 Claude 的 `system` 独立字段、Google 的 `contents` 结构）在此模块内部处理。

#### 流式传输

开启流式传输时，`onChunk` 回调逐块接收文本片段。主控制器将这些片段**实时渲染到聊天区域**（与酒馆原生流式体验一致），用户可看到文字逐字出现。流式渲染仅对写入聊天记录的模式生效（独立楼层 / 追加 / 覆写）；「仅解析」模式下流式数据仅在内存中累积，不做实时显示。

### 模块 4：后处理器

> 支撑功能：C-1 ~ C-2（写入方式、裁剪、推理剥离）

#### 职责

对 API 返回的原始文本执行后处理链，并按配置的写入方式路由到对应的输出方式。

#### 后处理链（固定顺序）

```text
1. 推理过程剥离
   └─ 若 Agent 配置未开启「传回推理过程」→ 丢弃 reasoning 字段

2. 裁剪开始标签前（C-2a）
   └─ 若启用 → 查找 startTag 首次出现位置，删除其及之前内容

3. 裁剪结束标签后（C-2b）
   └─ 若启用 → 查找 endTag 最后出现位置，删除其及之后内容
```

#### 写入方式路由

| 写入方式 | 实现 |
|----------|------|
| 仅解析 | 不操作 chat[]，仅通过事件广播消息内容 |
| 独立楼层 (User) | 调用酒馆 API 插入一条 `is_user: true` 的新消息 |
| 独立楼层 (Assistant) | 调用酒馆 API 插入一条 `is_user: false` 的新消息 |
| 追加到最新楼层末尾 | 读取 `chat[chat.length - 1].mes`，拼接后写回（可选前置换行） |
| 覆写最新楼层 | 直接替换 `chat[chat.length - 1].mes` |

所有写入方式都广播消息完成事件（模块 7），确保 VarUpdate 等监听方能收到内容。

**中间步骤的可见性**：写入聊天记录的步骤（独立楼层 / 追加 / 覆写）产生的消息会出现在 `chat[]` 中，后续 Agent 步骤在组装提示词时**自然可见**。若用户希望某些步骤的输出对后续步骤不可见，可通过正则规则的标签过滤机制将相关内容在提示词层面隐藏（仅隐藏提示词，不影响 chat 原始数据和 UI 显示）。

### 模块 5：重试控制器

> 支撑功能：D-1 ~ D-4（重试策略）

#### 职责

包裹模块 3（API 调用）和模块 4（后处理），在外层执行重试循环。

#### 重试判定流程

```text
调用 API + 后处理
  → D-1: 结束标识符检查
     ├─ 启用 → 输出中包含指定字符串？
     │   ├─ 否 → 判定失败，进入重试
     │   └─ 是 → 通过
     └─ 未启用 → 跳过

  → D-2: 变量解析联动
     ├─ 启用 → 广播消息完成事件 → 等待 VarUpdate 反馈
     │   ├─ 收到 varupdate:updated → 通过
     │   ├─ 收到 varupdate:update_failed → 广播 varupdate:retry_requested → 重试
     │   └─ 超时未收到 → 按 D-4 超时处理
     └─ 未启用 → 跳过

  → 所有检查通过 → 返回结果
  → 任一检查失败 → 重试计数 +1
     ├─ 未超过最大次数 → 等待重试间隔 → 重新调用 API
     └─ 已超过最大次数 → 暂停管道，弹出用户提示
```

### 模块 6：配置存储

### 模块 7：事件总线

> 支撑功能：G-1（信息输出）、C-1（消息完成广播）、D-2（VarUpdate 联动）

#### 职责

封装酒馆助手事件 API，定义 Agents 脚本的事件协议。

#### 事件定义

##### Agents 广播的事件

| 事件名 | 触发时机 | 说明 |
|--------|---------|------|
| `agents:pipeline_started` | 管道开始执行 | 通知所有监听方管道已启动 |
| `agents:step_started` | 单个 Agent 步骤开始 | 通知当前正在执行的步骤信息 |
| `agents:message_complete` | 单步生成完成（所有写入方式均触发） | **VarUpdate 主消息通道**。即使「仅解析」模式（不写入 chat[]），也通过此事件的 `content` 字段传递消息内容 |
| `agents:step_retry` | 单步触发重试 | 通知重试原因和当前重试计数 |
| `agents:step_failed` | 单步彻底失败 | 通知错误信息 |
| `agents:pipeline_ended` | 管道执行结束 | 通知管道最终结果 |
| `varupdate:retry_requested` | 重试控制器收到 `varupdate:update_failed` 后广播 | 通知 VarUpdate 回退变量状态，准备重新接收 |

##### Agents 监听的事件

| 事件名 | 来源 | 用途 |
|--------|------|------|
| `varupdate:updated` | VarUpdate | 重试控制器（D-2）：变量解析成功，通过判定 |
| `varupdate:update_failed` | VarUpdate | 重试控制器（D-2）：变量解析失败，触发重试 |

##### 事件负载类型定义

```typescript
/** agents:pipeline_started */
interface PipelineStartedPayload {
  schemeName: string;        // 当前方案名称
  agentCount: number;        // 启用的智能体数量
}

/** agents:step_started */
```

### 模块 8：通知系统

> 支撑功能：G-1 ~ G-3（通知等级、控制台输出格式）

#### 职责

统一的通知输出。面向用户功能卡定义了两类信息源——用户主动操作反馈（始终弹窗）和流水线自动事件（受通知等级控制）。

#### 通知等级

| 等级 | 数值 | 含义 |
|------|------|------|
| `debug` | 0 | 全部输出（含调试信息） |
| `always` | 1 | 通过+警告+错误 |
| `notice` | 2 | 警告+错误（默认） |
| `error` | 3 | 仅错误 |
| `silence` | 4 | 完全静默 |

#### 双通道输出

```typescript
/**
 * 流水线事件通知（受通知等级控制）
 */
function notify(level: NotifyLevel, title: string, message: string): void;

/**
 * 用户操作反馈（始终弹窗，不受通知等级限制，silence 除外）
 */
function feedback(success: boolean, title: string, message: string): void;
```

控制台输出格式：`[Agents] 步骤 A 完成 (2.3s, 1280 chars)`，按四色分级（🔵 debug / 🟢 通过 / 🟠 警告 / 🔴 错误）。

### 模块 9：UI 面板

> 支撑功能：I-1 ~ I-6（面板布局）、E-2（状态按钮）

#### 职责

脚本独立 UI 面板的渲染、事件绑定、拖拽排序、状态按钮管理。

#### 子组件

##### 9a. 主面板 (`panel.ts`)

对应 I-1 ~ I-4：
- 工具栏（导入/创建/内置资源按钮）
- 方案栏（切换/保存/导出/删除）
- 智能体排序区（工作区 + 待命区，支持拖拽）
- 全局设置栏（通知等级、排除策略、试生成提示词）

使用 jQuery + 原生 DOM 构建，注入到酒馆助手的脚本扩展面板容器。拖拽排序使用 HTML5 Drag and Drop API 或 SortableJS。

##### 9b. 智能体配置浮窗 (`agent-editor.ts`)

对应 I-5：顶部栏 + 左中右三栏布局的浮窗。

- 左栏：API 请求配置 + 重试策略
- 中栏：标签/历史层数 + 模型参数（根据 API 格式映射表动态显示）
- 右栏：后处理策略 + 备注
- 底部：Dry Run / 连接测试 / 试生成按钮（调用模块 10）

参数控件（滑动条、数值输入）的显示/隐藏由 `api-params-map.json` 驱动。

模型选取列表旁的「一键获取」按钮（A-4）：点击后调用模块 3 的 `fetchModelList()` 接口，传入当前智能体配置的 API 地址、Key 和格式，获取可用模型列表填充下拉框。请求期间按钮显示 loading 状态，失败时通过模块 8 的 `feedback()` 弹出红色错误提示。

##### 9c. 状态按钮 (`status-button.ts`)

对应 E-2：替代酒馆原生发送按钮。
- 空闲态：显示发送图标，点击启动管道
- 执行态：旋转动画 + 当前步骤字母，点击中止管道
- 实现：隐藏酒馆原生 `#send_but` / `#stop_but`，注入自定义按钮到相同位置

### 模块 10：辅助工具

> 支撑功能：H-1 ~ H-3（Dry Run、连接测试、试生成）

#### 职责

为 UI 面板的三个辅助功能按钮提供后端逻辑。

#### 接口

```typescript
/**
 * Dry Run：执行完整提示词组装但不发送 API
 * @returns 完整的 prompt 列表 + 预估 Token 数
 */
function dryRun(agentConfig: AgentConfig, labelCtx: LabelContext): Promise<DryRunResult>;

/**
 * 连接测试：向 API 发送轻量请求验证连通性
 */
function testConnection(url: string, apiKey: string, format: string): Promise<TestResult>;

/**
 * 试生成：使用测试提示词执行一次真实生成
 */
function testGenerate(agentConfig: AgentConfig, testPrompt: string): Promise<GenerateResult>;
```

- `dryRun` 调用模块 2（管道引擎）组装提示词，但不调用模块 3（API）。结果通过模块 8 的 `feedback` 输出。
- `testConnection` 调用 `fetchModelList`（模块 3）作为轻量探测。
- `testGenerate` 构建最小请求（仅测试提示词），调用模块 3 发送真实请求。

## 四、主控制器概述

主控制器（`index.ts`）是脚本的入口点和业务流程编排者。

### 初始化流程

```text
脚本加载
  → 模块 6（配置存储）初始化 extension_settings 连接
  → 模块 9（UI 面板）渲染主面板 + 自定义按钮
  → 模块 7（事件总线）注册酒馆事件监听
  → 模块 8（通知系统）初始化
```

### 管道执行流程

> **管道始终接管**：脚本加载后，所有生成请求都走多 Agent 管道。用户若需临时使用酒馆原生单次生成，可关闭脚本。单 Agent 方案（方案中仅一个智能体）等效于增强版单次生成。

```text
用户点击发送按钮（或脚本触发）
  → 模块 9: 切换为执行态按钮
  → 模块 7: 广播 agents:pipeline_started
  → 遍历当前方案的启用 Agent 列表（按排序）:
      → 模块 7: 广播 agents:step_started
      → 模块 2: 为当前 Agent 组装提示词
      → 模块 5: 重试循环包裹:
          → 模块 3: 发送 API 请求（流式时实时渲染到聊天区域）
          → 模块 4: 后处理 + 写入
          → 模块 5: 重试判定
      → 模块 7: 广播 agents:message_complete
      → 模块 8: 输出步骤完成通知
  → 模块 7: 广播 agents:pipeline_ended
  → 模块 9: 恢复空闲态按钮
```

### 脚本卸载流程

```text
监听 pagehide 事件
  → 模块 9: 恢复酒馆原生发送/中止按钮
  → 模块 7: 移除所有事件监听器
```
